Вимоги до змісту документу Vision & Scope:

Розділ 2. Описати призначення системи, вказати її головні функціональні можливості.
Розділ 3. Докладно описати функції кожної з частин системи та вимоги до цих частин.
Розділ 4. Описати усі зацікавлені у проєкті сторони. Вказати строки виконання проєкту, усі технології, які будуть використані у проєкті.
В цій лабораторній роботі необхідно зміни документ Vision & Scope, який було створено минулого семестру, а саме розділ 3: Scope and Limitations. В цьому розділі необхідно описати:
3.1 Scope of Initial Release — опис скоупу всього проекту разом із серверною частиною, мобільним і вебклієнтами та IoT. Описати роботу всієї системи.
3.2 Scope of Subsequent Releases — продумати, як у майбутніх релізах інтегрувати штучний інтелект. Треба описати 3 ключові аспекти роботи з ШІ:
* застосування — як ШІ може покращити роботу системи;
* збирання даних — які дані і яким чином можна збирати для подальшого навчання моделей;
* навчання/використання — чи треба навчати модель самостійно, донавчати існуючу, використовувати готовий API сервісу ШІ. Продумати, як краще за все використати дані, які можна збирати.
3.3 Limitations and Exclusions — з описаного функціоналу (з обох попередніх пунктів), які неочевидні обмеження. Який функціонал міг би бути присутній у системі, ал навмисно виключений і чому.

1. BUSINESS REQUIREMENTS (БІЗНЕС-ВИМОГИ)
1.1. Background (Передумови)
У сучасному агропромисловому секторі ефективне управління ґрунтовими ресурсами відіграє вирішальну роль для досягнення високої врожайності та сталого розвитку. Проте багато сільськогосподарських підприємств і фермерських господарств стикаються з проблемами:
•	відсутність оперативної інформації про стан ґрунту;
•	недостатня кількість даних для прийняття рішень щодо удобрення, поливу або сівозміни;
•	застарілі або ручні методи збору даних, які не забезпечують точності та регулярності.
Наявні системи агромоніторингу переважно не мають інтеграції з IoT-пристроями, працюють без географічної прив’язки до ділянок і не забезпечують зручний мобільний доступ до результатів аналізу. Це обмежує можливості швидкого реагування на зміни у ґрунтовому середовищі та призводить до неефективного використання ресурсів.
Запропонована система спрямована на вирішення наступних проблем:
•	Відсутність регулярного моніторингу вологості, pH та температури ґрунту.
•	Ускладнення у зборі та візуалізації польових даних.
•	Відсутність централізованої бази даних для аналізу та звітності.
•	Недостатня інтеграція з мобільними пристроями та персоналом у полі.

1.2. Business Opportunity (Бізнес-можливості)
Ринок агротехнологій активно розвивається, і попит на цифрові рішення для моніторингу ґрунту постійно зростає. У світі активно впроваджуються Smart Farming-підходи, однак в Україні та країнах Східної Європи попит все ще переважає над пропозицією доступних, локалізованих рішень.
Існуючі інструменти, як-от AgroMonitor або AgriSense, часто орієнтовані на великі агрохолдинги, мають високу вартість або обмежені функції у безкоштовних версіях. Більшість із них не пропонують офлайн-доступу, не підтримують інтеграцію з мобільними додатками чи локальними системами управління фермою.
Наша система — це можливість заповнити нішу рішенням, яке:
•	орієнтоване на малий та середній бізнес;
•	підтримує використання базових датчиків, які легко підключити та обслуговувати;
•	пропонує простий інтерфейс із мобільним доступом;
•	дозволяє візуалізувати та аналізувати ґрунтові дані в одному середовищі.

2. VISION OF THE SOLUTION (КОНЦЕПЦІЯ РІШЕННЯ)
2.1. Vision Statement (Окреслення концепції)
Програмна система моніторингу та аналізу ґрунту — це універсальне рішення для аграріїв, агрономів та науковців, яке дозволяє автоматично збирати, зберігати, аналізувати та візуалізувати дані про стан ґрунту. Система складається з IoT-модулів, вебінтерфейсу та мобільного додатку, що забезпечує повний контроль за змінами у ґрунтових характеристиках на конкретних полях у реальному часі.
Основні цілі:
•	Підвищення врожайності завдяки своєчасним діям на основі точних даних.
•	Оптимізація використання води та добрив.
•	Створення єдиного цифрового архіву даних про поля.
•	Надання інструментів для візуалізації, порівняння та звітування.





2.2. Major Features (Головна функціональність)
1.	MF-1: Моніторинг у реальному часі
o	Вимірювання вологості, температури та pH за допомогою IoT-датчиків.
o	Передача даних на сервер для централізованого збереження.
2.	MF-2: Система сповіщень
o	Повідомлення агроному або фермеру при досягненні критичних показників (наприклад, пересушений ґрунт).
o	Автоматичне інформування про необхідність поливу або коригування pH.
3.	MF-3: Формування звітів
o	Генерація історичних даних у вигляді таблиць та графіків.
o	Можливість експорту звітів у PDF або Excel.
4.	MF-4: Інтерактивна мапа полів
o	Відображення стану кожної ділянки з кольоровим кодуванням за параметрами ґрунту.
o	Географічна навігація між полями та зонами моніторингу.
5.	MF-5: Мобільний додаток для агронома
o	Швидкий доступ до інформації про ґрунт на конкретному полі.
o	Можливість введення ручних спостережень (фото, текст).
6.	MF-6: Налаштування системи
o	Конфігурація частоти зчитування показників.
o	Зміна порогових значень та алгоритмів сповіщення.
7.	MF-7: Синхронізація з агроінструментами
o	Можливість інтеграції з системами обліку посівів або поливу.
o	API для взаємодії з іншими агроплатформами.


3. SCOPE AND LIMITATIONS (РАМКИ ТА ОБМЕЖЕННЯ ПРОДУКТУ)
3.1. Scope of Initial Release (Рамки первинного випуску)
Перший випуск системи моніторингу та аналізу ґрунту включає повноцінну архітектуру: серверну частину, IoT-компоненти, вебінтерфейс для агрономів та мобільний застосунок для фермерів. Реалізовано функціонал збору даних, їх аналізу, візуалізації та оперативного сповіщення про зміни в параметрах ґрунту.
Серверна частина:
•	Обробка даних: реалізована на Flask (Python), забезпечує прийом, фільтрацію, зберігання й базовий аналіз даних від сенсорів.
•	REST API: дозволяє мобільному додатку й вебклієнту отримувати доступ до даних у реальному часі.
•	Сповіщення: при досягненні критичних параметрів (низька вологість, несподівані зміни pH) надсилаються повідомлення користувачам.
IoT-компоненти:
•	Сенсори: вологість, температура, pH, освітленість, інші — залежно від конфігурації.
•	Протоколи зв’язку: MQTT/HTTP для передачі даних на сервер.
•	Підключення до живлення або батарейна автономність.
Вебклієнт:
•	Вебпанель адміністратора для агронома/керівника:
o	Перегляд показників по полях;
o	Управління користувачами та ролями;
o	Експорт звітів (PDF, CSV).
•	Інтерактивна мапа з відображенням зон та кольоровим кодуванням показників.
Мобільний застосунок:
•	Для фермерів:
o	Перегляд стану ґрунту на полях у реальному часі;
o	Сповіщення про критичні показники;
o	Можливість додати вручну власні спостереження.
•	Для агрономів:
o	Огляд історичних змін;
o	Позначення ділянок для перевірки чи обробки;
o	Доступ до архіву показників за ділянками.
Безпека та захист даних:
•	HTTPS для передавання;
•	Шифрування на стороні сервера;
•	Бекапи та керовані політики зберігання;
•	2FA для адміністраторів.

3.2. Scope of Subsequent Releases (Рамки наступних випусків)
У майбутніх версіях заплановано:
•	Додавання нових типів сенсорів: наприклад, електропровідність, нітрати.
•	Підтримка супутникових знімків: для перехресної перевірки стану ділянок.
•	Мультиобліковий режим: для використання однієї фермерської групи кількома співробітниками з обмеженими правами.
•	Підтримка декількох господарств в одному акаунті: для агрохолдингів.
•	Інтеграція з системами автоматизованого поливу чи внесення добрив.
•	Офлайн-доступ до частини даних у мобільному додатку з подальшою синхронізацією.




3.3. Limitations and Exclusions (Обмеження та виключення)
Технічні обмеження:
•	Точність залежить від якості сенсорів і регулярності калібрування.
•	Відсутність гарантії зв’язку у віддалених районах без стабільного інтернету.
•	Наявність незначних затримок під час передачі великих обсягів даних.
Функціональні виключення:
•	Відсутність повної автоматизації дій (наприклад, запуск поливу).
•	Відсутність підтримки дуже старого або спеціалізованого обладнання без цифрових інтерфейсів.
•	Немає SMS- або соціальних сповіщень — лише push і email.
•	Відсутність відеоаналітики або розпізнавання зображень.
•	Обмежена кількість типів культур у базовій версії — оновлюється поступово.
Архітектурні обмеження:
•	Мобільний застосунок вимагає інтернету — офлайн-режим у первинній версії відсутній.
•	Підтримуються лише популярні мобільні ОС (Android 8+).
•	Збереження даних обмежується серверними потужностями — для великих господарств рекомендовано окремий сервер.

3.4 Operating Environment (Робоче середовище)
Система розробляється з використанням сучасних технологій, які забезпечують стабільність, масштабованість і легкість у використанні:
•	Backend: Flask (Python), REST API, обробка запитів і авторизація.
•	Database: Microsoft SQL Server або PostgreSQL — для зберігання даних від сенсорів та журналів активності.
•	IoT-симулятор: емуляція сенсорів для тестування функцій у лабораторних умовах без підключення реального обладнання.
•	Протоколи: MQTT або HTTP — залежно від типу мережі у господарстві.
•	Frontend: HTML + CSS + JavaScript (використання React.js або Bootstrap для адаптивності).
•	Mobile App: Kotlin (Android Studio) — синхронізується з сервером через REST API, підтримує push-повідомлення.
•	Безпека: використання HTTPS, шифрування даних, автентифікація користувачів.
Це середовище дозволяє адаптувати систему під різні типи користувачів: фермерів, агрономів, науковців та консультантів.
